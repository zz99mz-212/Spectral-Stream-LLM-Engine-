# Phase 2 Plan 4: Wire Model-Native Tokenizer into Eval (D-02 Gap Closure)

## Phase Goal

**As a** researcher running WikiText-2 perplexity, **I want to** specify `--tokenizer path/to/tokenizer.json` and have the eval actually use that model-native tokenizer for tokenization (not just record it as artifact metadata), **so that** absolute PPL values are faithful to the model's reference token distribution (D-02).

*(Derived from 02-VERIFICATION.md Truth #7, which is the single failing must-have from the Phase 2 verification report.)*

<objective>
Close Truth #7 from 02-VERIFICATION.md. Currently the eval's `--tokenizer` flag is recorded as artifact metadata only; it is never used for actual tokenization. The committed pre-tokenized sample uses byte-level IDs (not the model's SentencePiece/GGUF tokenizer), and raw-text fallback in `eval/corpus.py` uses `build_default_tokenizer()` (byte-level). The plan:

1. Wires `AutoTokenizer.from_pretrained` / `from_gguf` (via `auto_tokenizer_for_model`) into the actual tokenization path so the model's own tokenizer IS used when supplied.
2. Preserves the offline/reproducible default: when `--tokenizer` is omitted, `build_default_tokenizer()` remains the fallback so the committed byte-level sample continues to work.
3. Adds TDD tests proving the model tokenizer is called, not just recorded.
4. Documents the committed byte-level sample as the offline fallback (the model-native pre-tokenization can't be committed because the model file is git-ignored per D-12/D-13).
5. Updates the must_have language to accurately reflect what is now wired (D-02: model tokenizer used WHEN SUPPLIED, byte-level is the committed offline default).

Purpose: Closes the single remaining verification gap from Phase 2; makes the eval honest about its tokenizer provenance so absolute PPL values are comparable to community-standard implementations when the model tokenizer is provided.

Output: Patched `eval/corpus.py`, `eval/run_eval.py`, `eval/__init__.py`; new tests in `tests/test_eval_grader.py` proving the model tokenizer is actually used; a documentation comment in `eval/data/` clarifying the byte-level sample's role.

Out of scope (per D-02): no new pip dependencies (no `tiktoken`, `transformers`, `sentencepiece`); do not commit the actual model file; do not change D-07/D-08 (recovery ratio direction and threshold).
</objective>
